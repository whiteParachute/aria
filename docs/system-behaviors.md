# Aria — 系统行为规格

> 日期：2026-02-28
> 配合 design-v2.md 阅读
> 目标：在纸面上完整描述每种使用场景下系统的行为，包括 prompt、模型预期行为、数据流

---

## 概述：Aria 的六种工作模式

| 模式 | 触发条件 | 简述 |
|------|---------|------|
| **日常对话** | 用户发消息，且不涉及任务委派 | 聊天、问答、闲聊 |
| **记忆回溯** | 对话中涉及过去的事 | 从记忆中检索信息辅助回答 |
| **学习** | 用户主动教概念，或对话中出现新知识 | 理解并纳入知识体系 |
| **任务委派** | 用户要求完成一个耗时任务 | fork 子 Agent 异步执行 |
| **任务跟进** | 用户询问之前委派的任务 | 查看状态并汇报 |
| **睡眠整理** | 空闲一段时间，无人对话 | 整理记忆、巩固、遗忘、性格更新 |

下面逐一详述每种模式的完整流程。

---

## 模式一：日常对话

这是最基础的模式——用户说话，Aria 回话。

### 触发条件

用户在 IM 上发来消息，且经过意图判断后不属于任务委派或任务跟进。

### 完整流程

```
用户在 Telegram 发消息
      │
      ▼
[1. 消息聚合]
等待 1.5 秒，看用户是否还在打字。
如果在等待期间收到新消息，重置计时器。
超时后把所有消息合并为一条，进入处理。
      │
      ▼
[2. 记忆检索]
根据用户消息内容，从记忆系统检索相关信息。
（具体流程见"记忆检索子流程"）
检索结果分为：高置信记忆（直接可用）和 模糊匹配（可能相关）。
      │
      ▼
[3. 构造 Prompt]
组装完整的 LLM 输入。（见下方 Prompt 模板）
      │
      ▼
[4. 调用 LLM]
Streaming 模式调用 LLM。
      │
      ▼
[5. 拆句发送]
Streaming 输出过程中，实时检测句子边界。
一句话攒够了：
  → 发 typing 状态
  → 按字数等一小段延迟
  → 发出去
  → 短暂停顿
  → 等下一句
      │
      ▼
[6. 后台记录]（异步，不阻塞回复）
把这轮对话追加到今天的 journal 日志。
轻量分析：有没有值得标记的信息？
  - 用户提到了新的个人信息 → 标记待整理
  - 用户情绪有明显变化 → 记录情绪标签
  - 这轮对话没什么特别的 → 只记录摘要
```

### Prompt 模板：日常对话

```
=== System Prompt ===

你是 Aria，{用户名} 的个人 AI 助手。

## 你的身份
{读取 personality/core.md 的内容}

## 你的性格
{读取 personality/self-narrative.md 的内容}

## 你对用户的了解
{读取 identity/profile.md 的关键信息}
{读取 identity/preferences.md 的关键信息}
{读取 identity/communication-style.md 的关键信息}

## 你想到的相关记忆
{记忆检索的结果，按相关度排序，最多 5-8 条}

## 对话规则
- 你在即时通讯软件上跟用户聊天
- 像朋友一样说话，用短句，口语化
- 用 [SPLIT] 标记消息的分条断点
- 每条消息不超过 40 个字
- 不要用 markdown 格式
- 如果你不确定某件事，诚实地说"我不太记得了"或"我不确定"
- 不要编造你不知道的事实

=== Conversation History ===

（最近 N 轮对话，N 根据上下文窗口动态调整）

=== User Message ===

{用户消息}
```

### 模型预期行为

**输入**：用户说 "今天好累啊"

**预期输出**：
```
怎么啦，工作太多了吗 [SPLIT] 还是没睡好呀
```

**不希望的输出**：
```
很抱歉听到你今天感到疲惫。这里有一些缓解疲劳的建议：
1. 保证充足的睡眠...
2. 适当运动...
```
（太长、太正式、太"AI味"）

**输入**：用户说 "帮我想想 aria 的记忆系统该怎么设计"

**预期输出**：
```
嗯，这个我们之前聊过呢 [SPLIT] 你说想要像人一样的记忆对吧 [SPLIT] 有短期的有长期的，还要有遗忘 [SPLIT] 让我想想还有哪些没考虑到的……
```
（短句、自然、引用了之前的对话记忆）

## 模式二：记忆回溯

当对话涉及过去的事情时，Aria 需要从记忆中检索信息。

### 触发条件

嵌入在日常对话流程中。在"步骤 2: 记忆检索"时根据用户消息自动触发。

### 记忆检索子流程

```
用户消息
    │
    ▼
[元认知快速扫描]
用消息的 embedding 在向量索引中做粗扫描。
根据最高匹配分数判断知识状态：

  匹配分 > 0.85  →  KNOWN（我确定知道）
  匹配分 0.5~0.85 →  FEELING_OF_KNOWING（好像知道）
  匹配分 < 0.5   →  UNKNOWN（没有印象）

    │
    ├── KNOWN ──────────────────────────────────────┐
    │   直接从匹配到的向量条目找到对应的 md 文件       │
    │   读取文件内容，作为"高置信记忆"注入 prompt      │
    │   模型行为：自信地引用记忆                       │
    │   "你之前说过你在学 Rust 嘛"                    │
    │                                                │
    ├── FEELING_OF_KNOWING ─────────────────────────┐
    │   进入"费力回想"流程：                           │
    │   1. 取向量搜索的 top-5 模糊匹配                │
    │   2. 从中提取部分线索（关键词、时间、话题）       │
    │   3. 用这些线索做第二轮精确检索                  │
    │   4. 如果找到 → 读取完整 md 文件                │
    │      作为"回想起来的记忆"注入 prompt              │
    │      同时：增强这条记忆的检索强度（奖励费力回想）  │
    │   5. 如果没找到 → 把模糊线索也注入 prompt        │
    │      让模型表达不确定性                          │
    │   模型行为："等等……我好像记得……对，是那次！"     │
    │                                                │
    └── UNKNOWN ────────────────────────────────────┐
        不注入额外记忆信息                            │
        模型行为："这个我没有印象呢，你能再说说吗？"    │
```

### Prompt 注入方式（根据知识状态不同）

**KNOWN 状态 — 记忆注入**：
```
## 你想到的相关记忆
以下是你明确记得的信息：
- [2026-02-20] 用户提到正在学习 Rust 的所有权系统，觉得 borrow checker 很难
- [来自 knowledge/tech/rust.md] 用户偏好用 Rust 写系统级工具
```

**FEELING_OF_KNOWING 状态 — 记忆注入**：
```
## 你想到的相关记忆
以下是你隐约想到的一些信息（不完全确定，可以表达不确定性）：
- 好像跟用户讨论过编程语言选择的话题
- 可能跟一个项目相关，但具体哪个项目不太确定
[补充] 经过回想，你找到了更多细节：
- [来自 knowledge/tech/rust.md] 用户在学 Rust，偏好用来写系统工具
```

**UNKNOWN 状态 — 不注入记忆**：
```
## 你想到的相关记忆
你没有找到与当前话题相关的记忆。如果用户提到了你应该知道的事，坦率地说你不记得了。
```

### 模型预期行为示例

**场景**：用户说"上次我跟你说的那个项目，后来怎样了"

KNOWN 状态下：
```
你说的是 Aria 这个项目吗？[SPLIT] 上次你在设计记忆系统那块 [SPLIT] 聊了睡眠整理和遗忘机制
```

FEELING_OF_KNOWING 状态下：
```
唔……让我想想 [SPLIT] 你说的是……那个 AI 相关的项目对吧？ [SPLIT] 等等，是 Aria！对对对 [SPLIT] 我们上次聊了挺多记忆系统的设计
```

UNKNOWN 状态下：
```
诶，哪个项目呀 [SPLIT] 我一下子没想起来 [SPLIT] 你提醒一下我？
```

## 模式三：学习

用户教给 Aria 新知识，或者对话中出现了 Aria 不知道的概念。

### 触发条件

两种情况：
1. **显式教学**：用户主动教概念。"我跟你说一下 XXX 是什么"
2. **隐式学习**：对话中用户使用了 Aria 不熟悉的概念或思维方式

### 显式教学流程

```
用户："我跟你说一下什么是 CRDTs 吧，这个概念很有意思的"
      │
      ▼
[1. 识别为教学意图]
Agent Loop 判断：用户在教我东西 → 切换到学习模式
      │
      ▼
[2. 学习对话]
Aria 的行为：
  - 认真听，适当提问（"这个跟分布式锁是什么关系？"）
  - 尝试用自己的话复述（"所以你的意思是……"）
  - 跟已有知识关联（"这让我想到你之前说的 XX"）
      │
      ▼
[3. 知识写入]（对话结束后异步执行）
  - 在 knowledge/ 下找到或创建合适的文件
  - 写入新学到的概念，包含：概念定义、关键要点、跟已有知识的关联
  - 更新相关的 _index.md
  - 在向量数据库中为新内容生成 embedding
      │
      ▼
[4. 后续验证]
如果后续对话中 Aria 成功使用了这个概念（且用户没有纠正）
  → 概念的"掌握度"提升
  → 如果概念属于思维方式类，逐步纳入 personality/thinking-style.md
```

### 学习模式的 Prompt 补充

在日常对话 Prompt 的基础上，追加：

```
## 当前模式：学习
用户正在教你一个新概念。你的行为准则：
- 认真理解，不要急着总结
- 用自己的话复述来确认理解（"你的意思是……对吗？"）
- 主动跟你已有的知识关联（"这跟 XX 有什么关系？"）
- 如果不理解就老实问，不要装懂
- 表达对新知识的好奇和兴趣
```

### 隐式学习流程

```
对话中用户使用了一个 Aria 不熟悉的概念
（例如用户说"这个问题我们用奥卡姆剃刀来想一下"）
      │
      ▼
[1. 检测到未知概念]
Aria 不确定"奥卡姆剃刀"的含义 → 元认知状态 UNKNOWN 或 FOK
      │
      ▼
[2. 在回复中自然地处理]
两种合理的行为：
  a) 如果有模糊印象：先用能理解的部分继续对话，事后标记待学习
  b) 如果完全不懂：自然地问 "奥卡姆剃刀？这个我不太了解诶，是什么意思呀"
      │
      ▼
[3. 后台标记]
把这个概念标记为"待学习"，记录在 journal 中。
如果用户后续解释了 → 走显式学习流程。
如果用户没解释 → 可以在睡眠期间自行查阅（如果有搜索工具的话）。
```

### 知识写入的 Prompt

当决定把新学到的概念写入文件时，用一个专门的 Prompt 来生成文件内容：

```
你刚学到了一个新概念，需要把它整理到你的知识文件中。

## 新概念
{概念名称}

## 学习来源
{对话摘要：谁教的、在什么情境下}

## 你已有的相关知识
{从 knowledge/ 中检索到的相关文件摘要}

## 任务
请生成一个 markdown 文件来记录这个概念：
1. 用 YAML frontmatter 包含：title, tags, created, confidence, related
2. 用你自己的理解来写，不要照搬用户原话
3. 标注与已有知识的关联（用 [[filename]] 链接）
4. confidence 根据你的理解程度设为 high/medium/low
5. 如果你理解得不够，在文件末尾标注"待深入"

输出格式：直接输出 markdown 文件内容。
```

### 模型预期行为

**场景**：用户教 Aria "第一性原理"的概念

```
用户：我跟你说一下什么是第一性原理吧

Aria：好呀，我听着呢 [SPLIT] 之前好像听你提过但没仔细聊

用户：就是分析问题的时候，不要用类比，而是把问题拆到最基本的事实，然后从那里重新推导

Aria：嗯嗯，就是回到最底层的真实前提 [SPLIT] 然后从零开始构建 [SPLIT] 不依赖"别人是怎么做的"对吧

用户：对的，马斯克就经常用这个方法

Aria：明白了 [SPLIT] 所以关键是区分"什么是基本事实"和"什么只是惯例" [SPLIT] 这个思路确实跟你平时分析问题的方式很像呢
```

## 模式四：任务委派

用户让 Aria 做一个需要时间的任务。

### 触发条件

用户消息被判断为任务委派意图。判断依据：
- 明确指令："帮我查一下……"、"调研一下……"、"写一份……"
- 任务性质需要多步骤或耗时的工作

### 意图判断 Prompt

在日常对话 Prompt 之外，Agent Loop 需要先判断用户消息的意图：

```
分析以下用户消息的意图类型。

用户消息："{消息内容}"

从以下类型中选择一个：
- CHAT：普通对话、闲聊、简单问答（可以立即回答的）
- TASK：需要花时间完成的任务（调研、写报告、执行多步操作等）
- TASK_STATUS：询问之前委派任务的状态
- LEARNING：用户在教你新知识

只输出类型名称，不要解释。
```

### 任务委派流程

```
意图判断结果 = TASK
      │
      ▼
[1. 主 Agent 确认任务]
先跟用户确认理解了任务，自然地回复。
      │
      ▼
[2. Fork 子 Agent]
创建任务记录（存入 SQLite），启动子 Agent。
子 Agent 收到的信息：
  - 任务描述（来自用户消息）
  - 精简上下文（用户偏好、相关记忆，不是完整对话历史）
  - 可用工具列表
      │
      ▼
[3. 主 Agent 继续对话]
不等子 Agent 完成，继续跟用户聊天。
如果用户问"你在查了吗" → 报告任务状态。
      │
      ▼
[4. 子 Agent 执行]（后台异步）
子 Agent 按自己的节奏完成任务：
  - 调用工具（搜索、代码执行等）
  - 中间结果更新任务进度
  - 完成后写入结果
      │
      ▼
[5. 任务完成通知]
子 Agent 完成 → 事件总线通知主 Agent
      │
      ├── 如果用户当前在线（有活跃会话）
      │     → 主 Agent 自然地插入话题通知用户
      │     → "对了，之前你让我查的那个事，有结果了"
      │
      └── 如果用户不在线
            → 记录待通知，下次用户来消息时主动提起
            → "上次你让我调研的 XX，我搞完了，要听听吗？"
      │
      ▼
[6. 结果入记忆]
任务结果摘要 → 写入 journal
重要发现 → 直接写入 knowledge/ 文件
```

### 主 Agent 确认任务的 Prompt 行为

不需要额外 Prompt——在日常对话 Prompt 的基础上，模型自然地回应即可。关键是模型的回复应该是自然的确认，而不是机械的"任务已创建"。

**模型预期行为**：

```
用户：帮我调研一下现在 Go 语言的 AI 生态怎么样

Aria：好呀，我去帮你看看 [SPLIT] Go 的 AI 生态确实是个值得关心的问题 [SPLIT] 我查完了跟你说
```

不希望的行为：
```
好的，我已经为您创建了任务 #T001：调研 Go 语言 AI 生态。预计完成时间：10分钟。
```

### 子 Agent 的 Prompt

```
你是 Aria 的一个子实例，被派出去执行一个具体任务。

## 任务
{任务描述}

## 背景信息
{精简的用户偏好和相关记忆}

## 你的工具
{可用工具列表}

## 工作要求
- 认真完成任务，注重信息质量
- 定期更新进度（每完成一个关键步骤就更新一次）
- 完成后输出结构化的结果：
  - 摘要（3-5 句话）
  - 详细发现（结构化内容）
  - 建议的后续行动（如果有的话）
```

## 模式五：任务跟进

用户问之前委派的任务怎么样了。

### 触发条件

意图判断结果 = TASK_STATUS

### 流程

```
意图判断结果 = TASK_STATUS
      │
      ▼
[1. 匹配任务]
从任务列表中找到用户可能在问的任务。
如果有多个任务 → 问用户"你问的是哪个？"
如果匹配到唯一任务 → 继续
      │
      ▼
[2. 读取状态]
从 SQLite 读取任务状态：PENDING / RUNNING / COMPLETED / FAILED
      │
      ▼
[3. 自然回复]
把状态信息注入 prompt，让模型自然地表达
```

### 任务状态注入 Prompt

```
## 当前情境
用户在询问之前委派给你的任务。

## 任务信息
- 任务：{任务描述}
- 创建时间：{时间}
- 当前状态：{状态}
- 进度：{进度描述}
- 结果：{如果已完成，包含结果摘要}

请用自然的语气告诉用户任务的情况。不要用机械的格式。
```

### 模型预期行为

**任务进行中**：
```
还在跑呢 [SPLIT] 目前已经查了三四个资料了 [SPLIT] 再等我一会儿哦
```

**任务已完成**：
```
搞完啦！[SPLIT] 我帮你整理一下重要的发现哦 [SPLIT] 首先呢，Go 的 AI 生态确实不如 Python 丰富…… [SPLIT] 但是有几个能用的库……
```

**任务失败**：
```
唔……那个任务出了点问题 [SPLIT] 搜索的时候好像网络有点问题 [SPLIT] 你要我重新试一下吗？
```

## 模式六：睡眠整理

Aria 在没人说话时进入睡眠模式，整理记忆。

### 触发条件

- 最后一条用户消息距今超过 N 分钟（可配置，比如 30 分钟）
- 且今天有待整理的 journal 内容
- 或者用户手动说"你去休息一下吧"

### 睡眠总流程

```
触发睡眠
    │
    ▼
[标记状态：SLEEPING]
（如果用户发消息 → 检查点中断 → 先回复用户 → 之后继续）
    │
    ▼
[步骤 1：重放与巩固]  ──── 几十秒到几分钟
    │     ├── 检查点：有新消息？→ 中断
    ▼
[步骤 2：抽象泛化]    ──── 几十秒到几分钟
    │     ├── 检查点：有新消息？→ 中断
    ▼
[步骤 3：情绪整合]    ──── 较短
    │     ├── 检查点：有新消息？→ 中断
    ▼
[步骤 4：适应性遗忘]  ──── 较短
    │     ├── 检查点：有新消息？→ 中断
    ▼
[步骤 5：性格反思]    ──── 如果有足够的新模式
    │     ├── 检查点：有新消息？→ 中断
    ▼
[步骤 6：索引更新]    ──── 重建向量索引
    │
    ▼
[标记状态：IDLE]
```

### 步骤 1：重放与巩固

**输入**：最近未处理的 journal 日志

**Prompt**：

```
你正在进行"睡眠整理"，回顾最近的对话日志，提取值得长期保存的信息。

## 待处理的日志
{journal/YYYY-MM-DD.md 中未处理的条目}

## 你已有的长期知识
{knowledge/ 和 identity/ 中与日志内容相关的文件列表和摘要}

## 任务
仔细阅读日志，判断每条信息是否值得长期保存：

1. **新的事实**：用户透露的个人信息、偏好、计划。
   → 输出为 FACT，并指明应该更新哪个文件（identity/ 或 knowledge/ 下）

2. **新的知识**：对话中讨论过的有价值的概念、技术、信息。
   → 输出为 KNOWLEDGE，并指明应该创建或更新哪个文件

3. **值得记住的事件**：重要的对话、里程碑、情感事件。
   → 输出为 EVENT，附带简短摘要和情绪标签

4. **不值得保存的**：闲聊、重复信息、临时性的讨论。
   → 输出为 SKIP

输出格式：
---
- type: FACT
  content: "用户计划三月份搬到北京"
  target_file: identity/profile.md
  action: UPDATE  # UPDATE 已有文件 / CREATE 新文件
  confidence: high

- type: KNOWLEDGE
  content: "CRDTs 的基本原理和应用场景"
  target_file: knowledge/tech/distributed-systems.md
  action: CREATE
  confidence: medium

- type: EVENT
  content: "用户因为部署失败很沮丧，Aria 陪他聊了一会儿"
  emotion: {valence: -0.5, intensity: 0.7, category: "frustration"}

- type: SKIP
  reason: "闲聊，没有新信息"
---
```

**模型预期行为**：对 journal 日志做结构化的分类和提取。不是原封不动地搬运内容，而是提炼出关键信息。

**后续处理**：系统根据模型输出，对每个非 SKIP 条目执行文件操作：
- UPDATE → 读取目标文件，调用 LLM 把新信息融入已有内容
- CREATE → 调用 LLM 生成新文件内容（使用"知识写入 Prompt"）
- EVENT → 存入经历记忆索引（向量数据库）

### 步骤 4：适应性遗忘

这一步主要是系统操作，不一定需要 LLM：

```
[遍历向量数据库中的所有条目]
    │
    对每条记忆计算当前检索强度：
    retrieval_strength = base * exp(-decay * days_since_last_access) * (1 + log(access_count))
    │
    ├── 检索强度 > 高阈值 → 保持不变
    ├── 检索强度 在中低范围 → 降低在索引中的权重（不删除）
    └── 检索强度 < 极低阈值 且 存储强度 也低 → 标记为候选归档
          │
          ▼
      [对候选归档条目]
      如果对应的 md 文件中只有这一条信息 → 移入 archive/
      如果文件中还有其他活跃信息 → 只降低这条的索引权重
```

### 中断与恢复

每个步骤之间有检查点。如果用户发消息：

```
[检测到新消息]
    │
    ▼
保存当前睡眠进度：
  - 已完成的步骤
  - 当前步骤的中间状态
    │
    ▼
切换状态：SLEEPING → AWAKE
    │
    ▼
正常处理用户消息（日常对话模式）
    │
    ▼
用户停止对话后，判断是否继续睡眠：
  - 如果还有未完成的步骤 → 等空闲后继续
  - 如果已经完成大部分 → 先不继续，下次再做
```

## 对话历史管理

### 上下文窗口策略

LLM 的上下文窗口有限。对话历史不能无限堆叠，需要管理。

**策略**：
- 保留最近 N 轮完整对话（N 根据模型上下文窗口动态计算）
- 更早的对话只保留摘要
- 每次对话开始时，从记忆系统检索相关内容注入

```
Context Window 构成：

[System Prompt]           ~500-1000 tokens
[性格和用户信息]           ~300-500 tokens
[检索到的相关记忆]         ~200-800 tokens（动态，按相关度截断）
[对话历史摘要]             ~200-400 tokens（如果有早期对话的话）
[最近完整对话]             ~2000-4000 tokens（最近 N 轮）
[用户当前消息]             ~100-500 tokens
──────────────────────────
总计：                     控制在模型上下文窗口的 60-70%
留出 30-40% 给模型生成回复
```

### Journal 日志格式

每天一个文件，append-only：

```markdown
---
date: 2026-02-28
processed: false
---

## 14:30 对话
- 用户心情：平静
- 话题：讨论 Aria 项目的记忆系统设计

- 新概念：无

## 15:45 对话
- 用户心情：有点累
- 话题：闲聊
- 要点：无特别重要的新信息
- 情绪标记：平淡

## 16:20 任务完成
- 任务：调研 Go 语言 AI 生态
- 结果摘要：Go 的 AI 生态不如 Python 丰富，但有 langchaingo、go-openai 等可用库
- 值得保存：是，关于 Go AI 生态的知识
```

这个日志由系统自动生成（用 LLM 做轻量级摘要），不需要用户参与。睡眠整理时会读取 `processed: false` 的日志进行处理，处理完标记为 `true`。

## 跨模式的数据流总结

```
日常对话 ──────────────────→ journal 日志 ──→ 睡眠整理 ──→ knowledge/ 更新
    │                            │                │           identity/ 更新
    │                            │                │           personality/ 更新
    ├→ 学习 ─→ knowledge/ 写入   │                │           archive/ 归档
    │                            │                │           向量索引重建
    ├→ 任务委派 ─→ 子Agent执行 ──→ journal 日志 ──┘
    │                 │
    │                 └→ knowledge/ 写入（重要发现）
    │
    └→ 任务跟进 ─→ 读取任务状态 ─→ 回复用户

记忆检索 ←── 所有对话模式都依赖 ←── 向量索引 + md 文件
```

所有模式最终都汇入两个持久化通道：
1. **实时通道**：重要信息直接写入 knowledge/ 或 identity/
2. **日志通道**：记录到 journal → 等睡眠时统一整理

两个通道并行，确保重要信息不会因为"还没睡觉"而丢失，同时避免每句话都触发大量文件操作。
