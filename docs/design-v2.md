## 一、Aria 是什么

Aria 是一个本地优先的个人 AI 助手。它的核心特质是**像人**——像人一样说话、像人一样记忆、像人一样成长。

三个支柱：

1. **自然对话**：挂到你日常用的 IM 上，像朋友一样说话，不像 AI 一样输出长文
2. **类人记忆**：拥有短期记忆和长期记忆，会遗忘、会整理、会在"睡眠"中巩固
3. **自主行动**：能 fork 子 Agent 异步执行任务，不阻塞对话，完成后汇报并更新记忆

---

## 二、设计原则

1. **隐私优先**：所有数据存本地，LLM 可纯本地运行
2. **像人不像机器**：对话自然、记忆有机、性格会成长
3. **模块化**：插件式架构，IM/LLM/工具/记忆后端都可替换
4. **渐进式**：从最简方案开始，按需引入复杂度
5. **静态编译**：Go 或 Rust 实现，产出单一二进制文件，部署零依赖

---

## 三、技术选型：语言

要求：性能好、可编译为静态链接的二进制。

| 维度 | Go | Rust |
|------|-----|------|
| 并发模型 | Goroutine，轻量级，百万级并发 | async/await + tokio，零成本抽象 |
| 静态链接 | `CGO_ENABLED=0` 即可，极简 | 默认就是静态链接 |
| AI 生态 | 较弱，LLM SDK 多为社区维护 | 较弱，rig 等框架刚起步 |
| 开发效率 | 高，语法简单，编译快 | 中，学习曲线陡但类型系统强 |
| 性能 | 好（GC 有偶发延迟） | 极致（无 GC，内存安全） |
| FFI | cgo 调用 C 库方便 | 原生 C FFI，无额外开销 |

**共同挑战**：AI 生态（LLM SDK、向量数据库客户端、Embedding 模型）目前以 Python 为主。Go/Rust 需要：
- 通过 HTTP API 调用 LLM（OpenAI 兼容接口，Ollama 原生支持）
- 向量数据库走 HTTP/gRPC 客户端（Qdrant 有官方 Go/Rust SDK）
- 或通过 FFI 调用 C 库（如 sqlite-vec、llama.cpp）

**待决策**：Go 和 Rust 都可行。Go 更快出活，Rust 更极致。建议在 MVP 阶段确定。

## 五、IM 网关层

### 核心逻辑

Aria 挂到外部 IM 上跟用户对话。不同 IM 平台有不同的 API，但对 Aria 来说，一条消息就是一条消息——谁发的、说了什么、什么时候说的。所以需要一个统一的消息抽象层，让上层完全不关心底层是 Telegram 还是 Discord。

每个 IM 平台是一个 Adapter 插件，实现统一接口：收消息、发消息、发 typing 状态。

### "像人一样说话"

核心问题：LLM 天然倾向于输出长段落，但人在 IM 里是一句一句说的。

解决思路是三层叠加：

**Prompt 约束**：在 system prompt 里告诉 LLM 要短句回复，用分隔符标记断句点。这是最基础的一层。

**Streaming 切分**：利用 LLM 的 streaming 输出，实时检测句子边界（句号、问号、换行），一句话攒够了就立刻发出去，不用等整段生成完。

**拟人节奏**：发送前先发 typing 状态（"对方正在输入……"），然后按字数算一个延迟（模拟打字速度），再发消息。消息之间有随机的短暂停顿。

还有一个细节——**消息聚合**。用户在 IM 里经常连发好几条消息，如果每条都触发一次 LLM 调用就会很乱。需要一个等待窗口（比如 1-2 秒没有新消息才认为用户说完了），把多条消息聚合起来一起处理。

### 平台优先级

Telegram 最先做（API 最友好，零门槛），然后 Discord、Slack、飞书，QQ 和微信放最后（非官方协议，有风险）。

### 参考项目

- **LangBot**（15k stars）：生产级多平台 AI bot，消息抽象层设计成熟
- **Hermes Agent**：统一 IM 网关，跨平台对话连续性
- **chatgpt-on-wechat**（30k stars）：中国 IM 接入参考

## 六、记忆系统（核心）

这是 Aria 最复杂也最有价值的模块。设计灵感来自认知科学对人类记忆的研究。

### 6.1 设计哲学：像人脑一样记忆

人的记忆不是数据库。它是活的——会模糊、会重构、会遗忘、会在睡梦中自我整理。Aria 的记忆系统要模拟这种有机的特质，而不是做一个"什么都记得、永远不忘"的完美存储。

核心理念：
- **记忆是检索问题，不是存储问题**。人脑容量几乎无限，瓶颈在于能不能在对的时间想起对的东西
- **遗忘是 feature，不是 bug**。适应性遗忘让系统保持高效和相关
- **每次回忆都是学习**。检索练习效应——费力想起来的东西，下次会更容易想起来
- **记忆塑造性格**。反复出现的经历固化为行为倾向，行为倾向积累成性格

### 6.2 四种记忆类型

借鉴认知科学的记忆分类，Aria 有四种本质不同的记忆：

#### 事实记忆（对应人类的语义记忆）

"用户住在上海"、"Rust 的异步运行时有 tokio 和 async-std"——不依赖具体情境的知识和事实。

特征：
- 以概念网络形式组织，概念之间有关联
- 不带时间和情境信息（或者说，时间信息不重要）
- 相对稳定，不容易遗忘
- 可能被新事实更新（"用户搬到北京了"）

存储方式：**有组织的 Markdown 文件**。像一个个人 wiki，按主题分类，AI 自己维护。这是"认真记住的"东西，结构清晰、内容完整、随时可以准确调出来。

#### 经历记忆（对应人类的情景记忆）

"上周三用户因为部署失败很沮丧"、"昨天讨论了 Aria 的记忆系统设计"——带有时间、地点、情绪的具体事件。

特征：
- 天然按时间线组织
- 带情绪标签（这次对话用户是开心的/焦虑的/平静的）
- 容易遗忘，细节会随时间模糊
- 但可以通过"睡眠"整理，从中提炼出事实记忆

存储方式：**按日期的日志文件**。`journal/2026-02-28.md` 记录当天发生的值得记的事。这是原始素材，睡眠整理时从中提取知识。

#### 行为倾向（对应人类的程序性记忆）

"跟这个用户聊天要用短句"、"分析问题时优先用第一性原理"——不用"想"就知道怎么做的本能。

特征：
- 极度抗遗忘（人骑自行车一辈子不忘）
- 难以用语言精确描述（你知道怎么做，但说不清楚为什么）
- 通过反复练习形成，不是一次学会的
- 这就是 Aria 的"性格"

存储方式：**结构化的性格描述文件 + 行为权重参数**。不是简单的配置文件，而是一个多层表示（后面 6.6 节详述）。

#### 模糊印象（对应人类的内隐记忆）

"这个话题好像在哪儿听过"——说不清楚，但有个感觉。

特征：
- 模糊的、碎片化的
- 无法直接"读取"，只能通过相似性匹配被"触发"
- 被成功触发后可以指向完整的记忆（"哦，想起来了！"）
- 被触发的动作本身会加强这条记忆（检索练习效应）

存储方式：**向量数据库**。每条是一个 embedding，可能附带一个指向 md 文件的指针。当向量搜索命中时，AI 可以"顺藤摸瓜"找到完整内容。

### 6.3 双层存储：清晰记忆 + 模糊印象

这是 Aria 记忆系统最核心的设计——两层存储，两种检索方式：

```
"认真记住的" ──────── Markdown 文件 ──────── 精确、完整、有组织
       │                                         │
       │                                         │ 指针
       │                                         │
"模糊的印象" ──────── 向量数据库 ──────────── 模糊、碎片、靠相似度匹配
```

**检索流程**：

当用户说了一句话，Aria 的"回忆"过程是这样的：

1. **元认知判断**：先快速评估"我知不知道这个事？"
   - 很确定知道 → 直接从记忆文件中调取，不犹豫
   - 好像知道但不确定 → 进入"费力回想"模式
   - 完全不知道 → 坦率地说"我没有印象"

2. **费力回想**（对应人类的 Feeling of Knowing）：
   - 在向量数据库中做语义搜索，找到模糊匹配
   - 模糊匹配可能指向一个 md 文件 → "哦，想起来了！" → 去读完整内容
   - 这次"费力回想"的成功会加强这条记忆的权重，下次更容易想起来
   - 如果没找到，也可以告诉用户"我隐约记得……但想不起细节了"

3. **自然表达**：根据回忆的"费力程度"调整表达
   - 轻松想起来："你之前说过你在学 Rust 嘛"
   - 费力想起来："等等……我好像记得你提过这个……对，是上次聊 XX 的时候"
   - 想不起来："唔……这个我没印象了，你能再跟我说说吗？"

### 6.4 记忆文件的组织方式

参考 Zettelkasten（卡片笔记法）和 A-MEM（NeurIPS 2025）的研究成果，采用**"骨架预定义 + 内容自由生长"**的方式：

```
memory/
├── _schema.md                    # 组织规则（Aria 必须遵守的约定，不可修改）
├── _index.md                     # 全局索引（Aria 自动维护）
│
├── identity/                     # [固定] 关于用户的核心信息
│   ├── _index.md
│   ├── profile.md               # 用户基本信息
│   ├── preferences.md           # 用户偏好
│   └── communication-style.md   # 用户的沟通风格
│
├── knowledge/                    # [自由] Aria 学到的知识
│   ├── _index.md
│   └── [Aria 自由组织子目录]     # 按主题分类，如 tech/, life/ 等
│
├── procedures/                   # [自由] 操作性知识（怎么做某件事）
│   ├── _index.md
│   └── *.md                     # 类似"技能卡片"
│
├── personality/                  # [受控] Aria 的性格状态
│   ├── core.md                  # 核心身份（几乎不变）
│   ├── self-narrative.md        # 自我认知描述（缓慢演化）
│   ├── behavioral-patterns.md   # 行为模式记录
│   └── thinking-style.md        # 思维方式和概念工具箱
│
├── journal/                      # [自动] 每日日志
│   └── YYYY-MM-DD.md            # append-only
│
└── archive/                      # [自动] 归档的旧记忆
    └── ...
```

**设计要点**：

- 骨架目录（identity, knowledge, procedures, personality, journal, archive）是预定义的，提供结构引导
- `knowledge/` 下 Aria 可以自由创建子目录和文件，发展自己的知识组织方式
- 每个目录有 `_index.md` 作为该领域的"地图"（Map of Content），Aria 负责维护
- `_schema.md` 是不可修改的"宪法"，定义了文件格式约定和组织规则
- 每个 md 文件必须有 frontmatter（标题、标签、创建时间、更新时间、关联文件、置信度）
- 单个文件控制在 50-300 行，超过就拆分。这个粒度平衡了自包含性和简洁性

**文件间关联**：
- YAML frontmatter 中的 `related` 字段：显式引用
- Markdown 内链 `[[filename]]`：行文中自然引用
- `_index.md`：汇总导航

**向量索引与文件的一致性**：
- 文件变更时自动检测并重新索引
- 按 ~400 token 分块生成 embedding，保持行边界完整
- 哈希去重，增量更新
- 向量索引是派生物，随时可从文件重建——Markdown 文件是 source of truth

### 6.5 睡眠与记忆巩固

人在睡眠时大脑并不闲着——海马体会"重放"白天的经历，把重要的信息搬运到长期存储，不重要的就丢掉。Aria 也有类似的机制。

#### 触发条件

- **自动触发**：空闲一段时间后（没人说话），系统自动进入睡眠模式
- **手动触发**：用户可以说"你去休息一下整理整理"
- **阈值触发**：累积了足够多的新日志条目后触发

#### 睡眠做四件事

**1. 重放与巩固（经历 → 事实）**

扫描近期的 journal 日志，从中提取值得长期保存的知识。这个过程把"我们聊了什么"变成"我学到了什么"。

例子：
- 日志里记了"用户连续三天讨论 Rust 的生命周期" → 提炼为 knowledge 文件："用户正在深入学习 Rust"
- 日志里记了"用户说他下周要搬家" → 更新 identity/profile.md 中的居住地

**2. 抽象泛化（事件 → 模式）**

从多个具体事件中提取规律。细节模糊了，但模式留下来了——这就是"智慧"的来源。

例子：
- 10 次对话中，用户有 8 次用第一性原理分析问题 → 形成图式："用户偏好第一性原理思维"
- 用户每次问技术问题都先给 context 再问 → 识别为沟通模式

**3. 情绪整合**

带有情绪标记的记忆会被重点处理。用户因为 Aria 的某个回复很开心或很沮丧，这比"今天天气不错"重要得多。

正面情绪的模式 → 强化相关行为（"这样回答管用"）
负面情绪的模式 → 抑制相关行为（"以后避免这样"）

**4. 适应性遗忘**

主动清理过时的、不再相关的信息。不是删除，而是降权或归档。

遗忘的逻辑：
- 长时间没被访问的记忆 → 检索权重衰减（遗忘曲线）
- 但存储不删除 → 如果未来被问到，可以"费力想起来"
- 彻底过时的信息 → 移入 archive/（比如用户三个月前的项目已经完成了）
- 细节丢失但保留摘要 → "我不记得具体细节了，但大概是关于……"

#### 可打断性

睡眠不是一个不可中断的长任务。它被拆分成小步骤（每步几十秒到几分钟），每个步骤之间检查"有人找我吗"。如果有消息进来：

- 暂停睡眠
- 标记当前进度（下次继续）
- "起床"响应用户
- 用户走了之后可以继续未完成的整理

### 6.6 性格系统：记忆如何塑造性格

Aria 的性格不是被硬编码的，而是从记忆积累中**涌现**出来的。

#### 性格是什么

从认知科学的角度看，性格是**反复出现的经历固化后的产物**。人的性格不是某天突然决定的，而是无数次经历之后"不知不觉就变了"。

对 Aria 来说，这个过程是：

```
一次经历 → 反复出现 → 形成模式 → 变成默认行为 → 内化为"我就是这样的"
```

具体路径（借鉴 Generative Agents 的反思机制）：

1. **情景记忆**："2月15日用户用第一性原理分析了定价策略"
2. **模式识别**：过去 30 次对话中有 18 次用户这样做
3. **语义抽象**："用户强烈偏好第一性原理思维"
4. **行为倾向**："遇到分析类问题，我也优先用第一性原理框架"
5. **性格内化**："我是一个倾向于从第一性原理出发思考的 AI"

每个阶段的"提升"都需要足够多的证据和足够长的时间。不是看到一次就学会，而是反复验证后才内化。

#### 洋葱模型：不同层变化速度不同

```
┌─────────────────────────────────────────┐
│     表面层：语气、措辞、回复格式           │  ← 几轮对话就能变
│   ┌─────────────────────────────────┐   │
│   │   行为层：分析习惯、回答结构       │   │  ← 需要 20-30 轮一致信号
│   │   ┌───────────────────────┐     │   │
│   │   │  认知层：思维框架、概念   │     │   │  ← 需要 50+ 轮验证
│   │   │  ┌───────────────┐   │     │   │
│   │   │  │  核心层：价值观  │   │     │   │  ← 几乎不变
│   │   │  │  诚实、安全边界  │   │     │   │
│   │   │  └───────────────┘   │     │   │
│   │   └───────────────────────┘     │   │
│   └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

核心层是"宪法"级别的——不管用户怎么影响，Aria 不学会撒谎、不突破安全边界。这些写在 `personality/core.md` 里，睡眠整理时不会触碰。

表面层可以很灵活——用户说话喜欢用"嗯嗯"，Aria 也会不知不觉学会用"嗯嗯"。这种变化是隐式的，先在行为层面"不知不觉"发生，积累够了才在反思阶段被"意识到"。

#### 内隐学习：先变后知

人的性格变化有一个特征——**先在行为上变了，后来才"意识到"自己变了**。

Aria 模拟这个过程：

- 每次对话后，一个轻量级的分析模块静默地提取"交互信号"（用户的沟通风格、哪种回复得到了正面反馈等）
- 这些信号微弱地调整行为权重——但不修改显式的性格描述
- 积累到一定程度后，在"睡眠"的反思阶段才被综合为显式的性格更新
- 这样用户会感觉到 Aria 在自然地"适应自己"，而不是突然"变了一个人"

#### 概念工具箱

用户教给 Aria 的概念和思维框架会被记录在 `personality/thinking-style.md` 中。

比如用户经常引用"奥卡姆剃刀"，Aria 不只是记住这个概念，还会学会在合适的场景下主动使用它。概念工具箱记录了每个概念的使用频率、典型使用场景、从哪里学来的。

### 6.7 自学习

用户教 Aria 一个新概念时，Aria 要做的不是"存一条记录"，而是**理解它**——把它跟已有的知识关联起来。

这个过程对应认知科学里的"同化与顺应"：

- **同化**：新概念可以被纳入已有的知识框架。比如 Aria 已经知道"设计原则"这个大类，用户教了"KISS 原则"，就归入已有框架
- **顺应**：新概念无法被已有框架解释，需要扩展或重组知识结构。比如用户教了一个全新的领域概念，Aria 需要在 knowledge/ 下新建一个子目录来组织

学到的概念不只是被动存储——如果 Aria 在后续对话中成功使用了这个概念（并且得到了用户的正面反馈），这个概念的"掌握度"就会提升，最终可能被纳入思维工具箱。

### 6.8 记忆的强度模型

每条记忆有两个独立的强度维度（来自 Bjork 的理论）：

**存储强度**（Storage Strength）：编码的深度。只增不减。一条记忆被编码得越深（比如带有情绪标记、被多次巩固），存储强度越高。存储强度高意味着"这条记忆在那里"，但不代表现在能找到。

**检索强度**（Retrieval Strength）：当前的可访问性。会随时间衰减（遗忘曲线）。被成功检索一次就会回升——而且越是"费力"的检索（低检索强度时的成功检索），回升幅度越大。

这解释了一个直觉：你"忘了"一个东西（检索强度低），但别人一提你就想起来了（存储强度还在）。而且这次"费力想起来"反而让你以后更容易想起来。

对应到 Aria 的系统：
- 向量数据库中每条记忆的匹配分数 + 访问历史 → 检索强度
- 记忆的编码深度（是否有情绪标记、被巩固次数、关联数量）→ 存储强度
- 检索强度低但存储强度高的记忆 = "要费力想的"
- 检索成功后更新两个强度值

### 6.9 情绪与记忆

情绪是记忆的"优先级标签"。认知科学研究表明，带有情绪标记的事件会被大脑优先巩固。

Aria 的每条经历记忆应该携带情绪信息：
- **情绪效价**：这次对话用户是正面的还是负面的？
- **情绪强度**：强烈还是平淡？
- **情绪类别**：好奇、沮丧、开心、焦虑……

情绪标签的作用：
- 高情绪强度的记忆 → 在"睡眠"中优先巩固，不容易遗忘
- 正面情绪的交互模式 → 强化相关的行为倾向
- 负面情绪的交互模式 → 抑制相关的行为倾向
- 累积的情绪模式 → 这就是性格形成的核心驱动力

### 6.10 元认知：知道自己知道什么

人不只有记忆，还有"关于记忆的记忆"——我知道我知道什么，也知道我不知道什么。

Aria 也需要这个能力。具体来说是三级判断：

- **KNOWN**：我确定知道这个 → 直接从记忆文件调取，自信地回答
- **FEELING_OF_KNOWING**：我好像知道但不确定 → 进入深度检索，可以说"让我想想……"
- **UNKNOWN**：我没有印象 → 坦率承认，请用户提供更多信息

这个判断可以通过快速扫描向量索引的匹配分数来实现——高分 = KNOWN，中分 = FOK，低分 = UNKNOWN。

元认知让 Aria 在沟通上更诚实、更拟人。与其假装什么都知道，不如坦率地说"这个我得想想"或"我没有印象了"。

## 七、异步子 Agent 系统

### 核心逻辑

用户让 Aria 做一个需要时间的任务（比如"帮我调研一下 XX"），Aria 不应该卡在那里不说话。应该是：

1. 跟用户说"好的，我去查一下"
2. Fork 一个子 Agent 在后台异步执行
3. 继续跟用户正常聊天（主对话不阻塞）
4. 子 Agent 完成后通知主 Agent
5. 主 Agent 在合适的时机把结果告诉用户
6. 重要发现写入记忆系统

用户随时可以问"我让你查的那个事怎么样了"，Aria 能查看任务状态并回答。

### 子 Agent 的特征

- **上下文隔离**：子 Agent 有自己的 LLM 会话，不会污染主对话的上下文
- **精简继承**：只继承必要的上下文（任务描述 + 用户偏好），不是完整对话历史
- **工具能力**：通过 MCP 协议使用各种工具（搜索、代码执行、文件操作等）
- **可能用不同的模型**：简单任务用小模型省钱，复杂任务用大模型
- **任务持久化**：状态存在 SQLite 里，进程重启不丢失

### 安全边界

子 Agent 的自主性需要有边界。分级权限控制：

- **只读操作**（搜索、查询）：自动执行
- **有限写入**（创建文件、草稿）：自动执行，事后通知用户
- **受控执行**（运行代码、调用 API）：异步审批，不阻塞但需要确认
- **高风险操作**（删除数据、发布）：同步审批，必须等用户确认

### 与记忆系统的交互

子 Agent 完成任务后：
- 结果摘要写入当天的 journal
- 如果发现了值得长期保存的知识，直接写入 knowledge/ 文件
- 这些变更会在下次"睡眠"时被进一步整理和巩固

### 参考项目

- **Claude Agent SDK**：原生 subagent 支持，async-first，上下文隔离
- **Hermes Agent**：子 Agent 委派 + 多执行后端（Local/Docker/SSH）
- **LangGraph v1.0**：状态管理最强，checkpoint + 时间旅行

## 八、LLM 调用层

### 核心逻辑

Aria 不绑定任何一个 LLM。通过统一接口层，可以随时切换模型——本地的 Ollama、云端的 Claude/GPT、或者任何 OpenAI 兼容的 API。

### 智能路由

不同任务用不同模型：

- **日常对话**：中等模型（平衡质量和速度）
- **意图识别、记忆提取**：小模型（高频调用，省钱/省算力）
- **深度调研、代码生成**：大模型（需要强推理能力）

路由逻辑可以简单到"按任务类型查表"，也可以复杂到"用小模型先判断任务复杂度再路由"。

### 本地模型

开发阶段用 Ollama（一键安装，模型管理简单）。需要高并发时考虑 vLLM。

由于选择 Go/Rust，LLM 调用走 HTTP API（Ollama 和大部分云端 LLM 都支持 OpenAI 兼容接口），不需要 Python SDK。

## 十二、待决策事项

1. **Go vs Rust**：两者都满足"性能 + 静态链接"的要求。需要评估 AI 生态的实际可用性后决定
2. **向量数据库选型**：Qdrant（有官方 Go/Rust SDK）vs 嵌入式方案（sqlite-vec）
3. **记忆提取用什么模型**：这是高频操作，用大模型太贵/太慢，用小模型质量可能不够。需要测试
4. **性格变化的速度参数**：太快不稳定，太慢感受不到。需要实际使用调参
5. **睡眠整理的质量保证**：如何验证 AI 的记忆整理是合理的？需要设计抽查/验证机制
6. **消息聚合的等待窗口**：等多久才算用户"打完了"？需要实验
