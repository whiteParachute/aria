# Aria 记忆系统 — 讨论纪要 v0.3（单会话简化版）

> 日期：2026-03-07
> 状态：基于 v0.2 简化，去掉多会话并发假设
> 前提：单会话，挂在 Claude Code（Aria）上

---

## 一、核心思路

与 v0.2 一致：不从零写，做一个 Memory Agent 挂在现有的 Claude Code（Aria）上。

**与 v0.2 的关键变化：**
- 去掉多会话并发 → 只有一个会话
- Memory Agent 不需要是全局单例 → 按需调用即可
- 睡眠从两级合并为一级
- 去掉群聊相关设计

**不变的：**
- 四层记忆模型
- 热度机制与 compact
- 可信度规则
- 性格演化
- 时间绝对化

---

## 二、双 Agent 架构

```
用户 ←→ 主 Agent（Aria）←→ Memory Agent ←→ 记忆存储
```

### 主 Agent（Aria）
- 负责跟用户对话
- **不直接写记忆**
- 每次对话自动加载随身索引
- 需要查记忆时向 Memory Agent 发起查询

### Memory Agent
- **按需调用**，主 Agent 需要时调用它，完成后返回结果
- **拥有记忆的写入权**
- 负责记忆的写入、查询、冲突检测、分类归档
- 通信方向：主 → Memory（查询/写入请求）→ 主（返回结果）

与 v0.2 的变化：不再是全局单例常驻进程。单会话下没有并发写入问题，不需要单例来串行化。Memory Agent 也不需要主动通知主 Agent——主 Agent 调用它时自然能拿到结果。

### 触发机制
- **主动触发**：主 Agent 调用 Memory Agent（查询或用户说「记住这个」）
- **被动触发**：会话结束时触发睡眠维护

---

## 三、四层记忆存储

### 第一层：随身索引
- **性质**：每次对话自动加载到 context 的小文件
- **内容**：不是知识本身，而是「我知道什么」的地图
- **容量**：~200 条上限，分区管理
- **规则**：只放索引和摘要，不放具体内容；哪怕用户说「一定要记住」，也只在这层写一行索引，详细内容放第二层

#### 分区设计

```
## 关于用户（长期，很少变动）
- Yuelun，偏好简洁，第一性原理思维
- 目前在写 Go，关注 DeFi + AI Agent
[上限：~30 条]

## 活跃项目/话题（中期，按热度升降）
- 记忆系统设计 [有详细笔记]
- 投资研究 [有详细笔记]
- DeFi 借贷 [有详细笔记]
[上限：~50 条]

## 重要提醒（时效性，过期自动清理）
- 用户 2026-03-14 ~ 2026-03-21 去东京出差
[上限：~20 条]

## 近期上下文（短期，每次睡眠刷新）
- 上次在讨论记忆系统的简化设计
- 待定：MVP 实现范围
[上限：~50 条]

## 备用空间
[~50 条，给溢出和临时提升用]
```

与 v0.2 的变化：近期上下文不再按会话分区（私聊/群聊A/群聊B），简化为一个扁平区域。

### 第二层：检索知识
- **性质**：按领域组织的详细知识库
- **内容**：结晶化的事实、用户档案、领域笔记
- **组织**：按领域/主题分目录，由 Memory Agent 自由管理
- **访问**：主 Agent 按需通过 Memory Agent 查询

### 第三层：模糊印象（语义索引文件）

与 v0.2 完全一致：

- **性质**：全局的语义索引，跨所有领域
- **作用**：当前两层都没命中时，提供「好像在哪儿见过」的模糊关联
- **实现**：Memory Agent 在睡眠时为每次会话生成一个**描述性索引文件**，包含话题摘要、关键词、涉及的人/事/概念。查询时用字符串匹配搜索这些文件，命中后再展开原始对话记录
- **组织**：按会话组织，每个会话一个索引文件
- **优势**：比向量数据库轻量，可读可调试，不需要额外基础设施
- **自我修复**：Memory Agent 在读新会话记录时，如果发现某条记忆被成功召回但索引文件中缺少对应的关键词/关联，会回去补充该索引文件。反之，如果发现误命中（搜到了但实际不相关），也会修正索引以减少噪音

语义索引文件示例：
```
# 2026-03-07 会话 — 记忆系统架构讨论

## 话题
- 讨论了 Aria 记忆系统的架构设计
- 四层记忆模型：随身索引、检索知识、模糊印象、原始记录
- 热度机制和 compact 策略
- Memory Agent 的触发方式（主动 + 被动）

## 关键词
记忆系统, 双Agent架构, 随身索引, compact, 热度, 性格演化

## 涉及的人/事/概念
- Qdrant（提了一嘴，没深入）
- Claude Code 作为主 Agent 的宿主
```

### 检索瀑布流程

```
用户问："我们是不是聊过 Qdrant？"

1. 主 Agent 扫随身索引 → 没有
2. 主 Agent 问 Memory Agent
3. Memory Agent grep 语义索引文件 → 命中 2026-03-07 摘要
4. Memory Agent 返回结果："有，2026-03-07 聊记忆系统时提过，当时在讨论向量数据库选型"
5. 如需更多细节 → Memory Agent 读原始记录补充
6. 主 Agent 用自己的语气转述给用户
```

与 v0.2 的变化：Memory Agent 不再「分步回复」（先模糊再补细节），而是一次性返回完整结果。主 Agent 可以在自己的回复中模拟回忆的过程感。

### 第四层：原始记录
- **性质**：对话日志，append-only
- **内容**：完整的对话历史
- **作用**：source of truth，其他三层都从这里派生

### 层间关系

与 v0.2 一致：
```
随身索引 ——→ "我知道我知道 X"（元认知地图）
    │
    ↓ 需要细节时
检索知识 ——→ "X 的具体内容是……"
    │
    ↓ 前两层没命中
模糊印象 ——→ "好像跟什么有关？" → 指向检索知识或原始记录
    │
    ↓ 最后的兜底
原始记录 ——→ 完整对话历史
```

## 四、热度机制

与 v0.2 完全一致：

知识在层间会**升降**：

- **升级**（检索 → 随身）：频繁使用、近期重要、用户强调
- **降级**（随身 → 检索）：长时间没访问，热度衰减
- **compact**：随身层满了 → 触发压缩（多条合并为一条、精简表述、降级不常用的）

compact 示例：
```
压缩前（三条）：
  - "用户在学 Go"
  - "用户之前学过 Rust 但放弃了"
  - "用户偏好静态类型语言"

压缩后（一条）：
  - "用户偏好静态类型语言，目前在写 Go（之前试过 Rust）"
```

### 访问追踪
- 「用到了」就算访问（不要求精确追踪）
- Memory Agent 通过读聊天记录来判断哪些知识被用到了
- 每条记忆带元数据：创建时间、最后访问时间、访问次数、来源人、关于谁、可信度标签

---

## 五、睡眠机制

### 定义
- **睡眠** = 用户结束一次会话到开启新会话之间的时间
- **会话结束** = 用户手动告知（说「先这样吧」/「晚安」等），或直接关闭会话
- 类比人类睡眠时大脑整理记忆的过程

### 维护流程（单级）

与 v0.2 不同，不再区分会话级收尾和全局睡眠。会话结束时统一执行：

1. 为本次会话生成**语义索引文件**（写入模糊印象层）
2. 提炼本次会话中的新知识 → 写入检索知识层
3. **近因提升**：把本次对话的重要内容升到随身索引的「近期上下文」区
4. **过期清理**：移除已过时的记忆（如已经过去的出差安排）
5. **compact**：随身索引超限 → 压缩合并、降级低热度条目
6. **自审**：检查记忆组织是否规整——
   - 有没有业务知识混进随身索引？
   - 有没有重复条目？
   - 分区比例是否合理？
7. **性格线**（不是每次都执行，每 N 次睡眠检查一次）：分析交互模式，必要时微调主 Agent 行为指令

---

## 六、异步维护的两条线

与 v0.2 一致：

### 知识线
原始记录 → 筛选值得记的 → 提炼为简短 fact → 定位领域 → 检查冲突 → 写入/合并/更新 → 热度评估 → compact

### 性格线
原始记录 → 提取交互模式 → 积累信号 → 达到阈值 → 修改主 Agent 的 system prompt / 行为指令

性格不是「知识」，而是对主 Agent 行为的直接修改。

---

## 七、硬规则

### 时间绝对化

与 v0.2 完全一致：
```
❌ "用户下周要去东京出差"
✅ "用户 2026-03-14 ~ 2026-03-21 去东京出差（2026-03-07 提及）"
```

记录时间（何时提及）和事件时间（何时发生）都要保留。

### 写入权
只有 Memory Agent 可以写入记忆存储。主 Agent 不直接修改记忆文件。

### 随身索引
只放索引，不放内容。超限触发 compact，不触发丢弃。

### 记忆可信度（自述优先原则）

与 v0.2 一致。单会话下暂时只有用户一个信息来源，但规则保留以备后续扩展多人场景。

| 来源 | 关于谁 | 可信度 | 处理方式 |
|---|---|---|---|
| 用户自己说 | 关于自己 | 完全可信 | 直接写入 |
| 用户自己说 | 关于别人 | 较可信（用户视角） | 写入，标注「用户认为」 |

---

## 八、同步 vs 异步写入

与 v0.2 一致：

| | 同步（用户说「记住这个」） | 异步（睡眠整理） |
|---|---|---|
| 触发 | 主 Agent 转发给 Memory Agent | 会话结束 |
| 筛选 | 跳过（用户已决定重要性） | 需要 LLM 判断什么值得记 |
| 定位 | 定位 + 写入 | 定位 + 写入 + 合并 |
| 热度 | 不需要评估 | 需要评估升降级 |
| 额外工作 | 无 | compact、性格演化、遗忘 |

## 九、用例推演

### 用例 1：跨会话事实记忆

**场景**：用户今天说「我下周去东京出差」。三周后说「好累啊」。

**流程**：
1. 对话中，这句话进入 → **原始记录**
2. Memory Agent 被动触发（对话结束）→ 提炼 fact「用户 2026-03-14 ~ 2026-03-21 去东京出差」
3. 写入 → **检索知识/用户/近期动态**
4. 睡眠时生成 → **语义索引文件**（模糊印象层）
5. 判断为近期重要事项 → 升到 **随身索引**：「用户 2026-03-14 ~ 2026-03-21 去东京出差」
6. 三周后用户说「好累」→ 主 Agent 随身索引里就有这条 → 直接脱口而出：「是不是东京出差累到了？」
7. 再过两个月没提 → 热度衰减 → 从随身索引降回检索知识

### 用例 2：模糊回想

**场景**：用户问「我们是不是聊过 Qdrant？」

**流程**：
1. 主 Agent 扫随身索引 → 没有 Qdrant 相关
2. 主 Agent 问 Memory Agent：「我们聊过 Qdrant 吗？」
3. Memory Agent grep 语义索引文件 → 命中 2026-02-20 的会话摘要（关键词里有 Qdrant）
4. Memory Agent 返回：「有，2026-02-20 聊向量数据库选型时提过，你在比较 Qdrant 和 Milvus，最后没选」
5. 主 Agent 用自己的语气回复用户

### 用例 3：性格适应

与 v0.2 一致：

**场景**：用户连续二十次对话都用简短风格。

**流程**：
1. 每次对话 → **原始记录**
2. Memory Agent 异步性格线 → 分析交互模式 → 「用户偏好简短对话」信号反复出现
3. 积累 20+ 次 → 达到阈值 → Memory Agent **修改主 Agent 的行为指令**：提高「回复简洁度」权重
4. 主 Agent 渐进地变得更简洁

### 用例 4：用户主动说「记住这个」

与 v0.2 一致：

**场景**：用户说「记住，我的服务器 root 密码是不能告诉别人的」

**流程**：
1. 主 Agent 识别到「记住」意图 → 主动触发 Memory Agent
2. Memory Agent 跳过筛选（用户已声明重要性）
3. 转为绝对时间 → 写入 **检索知识/用户/安全规则**
4. **随身索引** 加一行：「安全：服务器 root 密码不可外泄 [已记录]」
5. 不在随身索引里放密码本身

### 用例 5：记忆冲突

与 v0.2 一致：

**场景**：半年前说「在学 Rust」，现在说「在写 Go，Rust 放弃了」

**流程**：
1. 新对话 → **原始记录**
2. Memory Agent 异步触发 → 提炼 fact「用户在写 Go」
3. 检测到与已有 fact「用户在学 Rust」冲突
4. 更新 **检索知识**：标记旧条目为历史（「2025-12 ~ 2026-02 学过 Rust，后放弃」）
5. **随身索引** 更新：「用户目前在写 Go」

### 用例 6：领域知识学习

与 v0.2 一致：

**场景**：用户说「帮我把这篇 DeFi 借贷的文章学一下」

**流程**：
1. 这不走「经历 → 提炼」路线，而是 Memory Agent **直接写入检索知识** `知识/defi/借贷协议.md`
2. 睡眠时生成 **语义索引文件**，包含该学习内容的关键词和概念
3. **随身索引** 加一行：「领域知识：DeFi 借贷 [有详细笔记]」
4. 后续频繁聊 DeFi → 核心概念可能升到随身索引的摘要里
5. 长期不提 → 热度衰减，随身索引只保留一行索引，细节留在检索层

### 用例 7：随身索引 compact

与 v0.2 一致：

**场景**：随身索引接近容量上限。

**流程**：
1. Memory Agent 检测到随身索引即将超限
2. 扫描每条的元数据（最后访问时间、访问次数）
3. 合并相关条目：「用户在写 Go」+「用户之前学过 Rust」+「用户偏好静态类型」→「用户偏好静态类型语言，目前写 Go（曾学 Rust）」
4. 降级低热度条目 → 从随身索引移除，检索知识里还有
5. 精简表述 → 去掉冗余修饰词
6. 确保压缩后不丢失「我知道什么」的索引能力

## 十、已确定事项

| 话题 | 结论 |
|---|---|
| 会话模型 | 单会话，不支持多会话并发 |
| Memory Agent 形态 | 按需调用，不是常驻单例 |
| 通信方向 | 主 → Memory → 主（单向调用+返回），无主动通知 |
| 模糊印象实现 | LLM 生成描述性语义索引文件（按会话），字符串匹配，命中再展开原始记录 |
| 搜索流程 | 随身索引 → 语义索引文件 → 原始记录 |
| 睡眠定义 | 会话结束即触发 |
| 睡眠分级 | 不分级，统一执行全部维护 |
| 随身索引容量 | ~200 条，分区管理（用户/话题/提醒/近期上下文/备用） |
| 近期上下文 | 扁平区域，不按会话分区 |
| 索引自我修复 | Memory Agent 检索时发现缺失/误命中 → 补充/修正 |
| 记忆可信度 | 自述优先原则，保留规则备扩展 |

### 与 v0.2 的差异汇总

| v0.2 | v0.3 | 原因 |
|------|------|------|
| 多会话并发（私聊 + 群聊） | 单会话 | 简化 |
| Memory Agent 全局单例 | 按需调用 | 无并发，不需要单例 |
| 双向通信（含主动通知） | 单向调用 + 返回 | 按需调用模式下不需要主动通知 |
| 两级睡眠（会话级 + 全局） | 一级睡眠 | 只有一个会话 |
| 近期上下文按会话分区 | 扁平区域 | 只有一个会话 |
| 分步回复（先模糊再补细节） | 一次性返回完整结果 | 按需调用模式的限制 |
| 群聊用例（用例 3） | 去掉 | 单会话 |
| 可信度五种来源组合 | 简化为用户自述两种 | 暂无第三方来源 |
