# Aria 记忆系统 — 讨论纪要 v0.2
> 日期：2026-03-07
> 状态：架构讨论，尚未定稿

---

## 一、核心思路转变

原设计文档（v2）规划了一个完整的 Aria 系统（IM 网关 + LLM 层 + 记忆 + 子 Agent），从零用 Go/Rust 实现。

讨论后的新方向：**不从零写，而是做一个 Memory Agent 挂在现有的 Claude Code（Aria）上**。

理由：
- Claude Code 已经是一个能用的对话 Agent，有子 Agent、工具调用、system prompt 等能力
- IM 网关、LLM 调用、子 Agent 管理这些全不用写
- 直接聚焦在最有价值的部分——记忆系统本身
- 可以快速验证记忆系统的设计是否真的好用

---

## 二、双 Agent 架构

```
会话 A（私聊）──→ ┐
会话 B（群聊1）──→ ├──→ Memory Agent（单例）──→ 记忆存储
会话 C（群聊2）──→ ┘
```

### 主 Agent（Aria）
- 负责跟用户对话
- **不直接写记忆**
- 每次对话自动加载随身索引
- 需要查记忆时向 Memory Agent 发起查询
- 可能同时存在多个会话实例（私聊、多个群聊）

### Memory Agent
- **全局单例**，所有会话共享同一个 Memory Agent
- **拥有记忆的写入权**
- 负责所有记忆的写入、整理、compact、热度管理、性格演化
- 双向通信：
  - Memory → 主：主动通知（"用户刚说要搬家，我已更新"）
  - 主 → Memory：查询（"我们聊过 Qdrant 吗？"）
- 作为唯一写入者，天然解决多会话并发写入冲突

### 并发模型
- 同一时刻可能有多个会话在跑（私聊 + 多个群聊）
- 记忆存储是所有会话共享的 → **知识自动跨会话流动**
  - 例：用户在私聊说"我下周去东京出差"，群里有人问用户在不在 → 群聊的主 Agent 扫随身索引就能看到这条
- Memory Agent 串行处理来自各会话的请求，避免写入冲突

### 触发机制
- **主动触发**：主 Agent 直接与 Memory Agent 对话（查询或用户说"记住这个"）
- **被动触发**：会话结束时触发（用户手动告知会话结束）

## 三、四层记忆存储

### 第一层：随身索引
- **性质**：每次对话自动加载到 context 的小文件
- **内容**：不是知识本身，而是"我知道什么"的地图
- **容量**：~200 条上限，分区管理
- **规则**：只放索引和摘要，不放具体内容；哪怕用户说"一定要记住"，也只在这层写一行索引，详细内容放第二层

#### 分区设计

```
## 关于用户（长期，很少变动）
- Yuelun，偏好简洁，第一性原理思维
- 目前在写 Go，关注 DeFi + AI Agent
[上限：~30 条]

## 活跃项目/话题（中期，按热度升降）
- 记忆系统设计 [有详细笔记]
- 投资研究 [有详细笔记]
- DeFi 借贷 [有详细笔记]
[上限：~50 条]

## 重要提醒（时效性，过期自动清理）
- 用户 2026-03-14 ~ 2026-03-21 去东京出差
[上限：~20 条]

## 近期上下文 — 按会话（短期，每次睡眠维护时刷新）
[上限：~50 条总量]

### 私聊（最近一次）
- 在讨论记忆系统架构设计
- 待定：MVP 范围、LLM 选型

### 群聊·技术群A（最近一次）
- 在聊 Go 的泛型设计
- 小明提了个有趣的 interface 用法

### 群聊·技术群B（最近一次）
- 不太活跃，上次是三天前

## 备用空间
[~50 条，给溢出和临时提升用]
```

每个活跃会话在"近期上下文"里有独立的小区域，切换会话时主 Agent 可以快速回忆上次聊了什么。近期上下文体现**近因效应**——距离近的记得牢，每次睡眠维护时刷新。

### 第二层：检索知识
- **性质**：按领域组织的详细知识库
- **内容**：结晶化的事实、用户档案、领域笔记、群成员信息等
- **组织**：按领域/主题分目录，由 Memory Agent 自由管理
- **访问**：主 Agent 按需通过 Memory Agent 查询

### 第三层：模糊印象（语义索引文件）
- **性质**：全局的语义索引，跨所有领域
- **作用**：当前两层都没命中时，提供"好像在哪儿见过"的模糊关联
- **实现**：Memory Agent 在睡眠时为每次会话生成一个**描述性索引文件**，包含话题摘要、关键词、涉及的人/事/概念。查询时用字符串匹配搜索这些文件，命中后再展开原始对话记录
- **组织**：按会话组织，每个会话一个索引文件
- **优势**：比向量数据库轻量，可读可调试，不需要额外基础设施
- **自我修复**：Memory Agent 在读新会话记录时，如果发现某条记忆被成功召回但索引文件中缺少对应的关键词/关联，会回去补充该索引文件。反之，如果发现误命中（搜到了但实际不相关），也会修正索引以减少噪音。类似人类的记忆强化——回忆一次后关联路径变多，下次更容易想起来

语义索引文件示例：
```
# 2026-03-07 会话 — 记忆系统架构讨论

## 话题
- 讨论了 Aria 记忆系统的架构设计
- 四层记忆模型：随身索引、检索知识、模糊印象、原始记录
- 热度机制和 compact 策略
- Memory Agent 的触发方式（主动 + 被动）

## 关键词
记忆系统, 双Agent架构, 随身索引, compact, 热度, 性格演化

## 涉及的人/事/概念
- Qdrant（提了一嘴，没深入）
- Claude Code 作为主 Agent 的宿主
```

### 检索瀑布流程

```
用户问："我们是不是聊过 Qdrant？"

1. 主 Agent 扫随身索引 → 没有
2. 主 Agent 问 Memory Agent
3. Memory Agent grep 语义索引文件 → 命中 2026-03-07 摘要
4. Memory Agent 先快速回复："好像有，是 3 月 7 号聊记忆系统的时候提过"
5. Memory Agent 再去读该会话的原始记录，补充细节
6. 如果跨多次会话 → Memory Agent 拼接相关片段，给出完整上下文
```

Memory Agent **分步回复**：先给模糊印象（"好像有"），再补充细节。更像人的回忆过程。

### 第四层：原始记录
- **性质**：对话日志，append-only
- **内容**：完整的对话历史
- **作用**：source of truth，其他三层都从这里派生

### 层间关系
```
随身索引 ——→ "我知道我知道 X"（元认知地图）
    │
    ↓ 需要细节时
检索知识 ——→ "X 的具体内容是……"
    │
    ↓ 前两层没命中
模糊印象 ——→ "好像跟什么有关？" → 指向检索知识或原始记录
    │
    ↓ 最后的兜底
原始记录 ——→ 完整对话历史
```

## 四、热度机制

知识在层间会**升降**：

- **升级**（检索 → 随身）：频繁使用、近期重要、用户强调
- **降级**（随身 → 检索）：长时间没访问，热度衰减
- **compact**：随身层满了 → 触发压缩（多条合并为一条、精简表述、降级不常用的）

compact 示例：
```
压缩前（三条）：
  - "用户在学 Go"
  - "用户之前学过 Rust 但放弃了"
  - "用户偏好静态类型语言"

压缩后（一条）：
  - "用户偏好静态类型语言，目前在写 Go（之前试过 Rust）"
```

### 访问追踪
- "用到了"就算访问（不要求精确追踪）
- Memory Agent 通过读聊天记录来判断哪些知识被用到了
- 每条记忆带元数据：创建时间、最后访问时间、访问次数、来源人、关于谁、可信度标签

## 五、睡眠机制

### 定义
- **睡眠** = 用户结束一次会话到开启新会话之间的时间
- **会话结束** = 用户手动告知（说"先这样吧"/"晚安"等），不做超时检测
- 类比人类睡眠时大脑整理记忆的过程

### 两级维护

#### 会话级收尾（轻量，某个会话结束时）
1. 为该会话生成**语义索引文件**（写入模糊印象层）
2. 提炼该会话中的新知识 → 写入检索知识层
3. 标记该会话为已整理

#### 全局睡眠（重量级，所有会话都结束时）
1. **近因提升**：把最近几次对话的重要内容临时升到随身索引的"近期上下文"区
2. **过期清理**：移除已过时的记忆（如已经过去的出差安排）
3. **compact**：随身索引超限 → 压缩合并、降级低热度条目
4. **自审**：检查记忆组织是否规整——
   - 有没有业务知识混进随身索引？
   - 有没有重复条目？
   - 分区比例是否合理？
5. **性格线**：分析交互模式，必要时微调主 Agent 行为指令

### 并发场景下的睡眠

- 某个会话结束 → 只做该会话的轻量收尾
- 最后一个会话也结束了 → 触发全局睡眠
- 好处：每个会话的素材及时整理，不浪费；全局维护等所有会话安静后再做，不干扰进行中的对话

## 六、异步维护的两条线

### 知识线
原始记录 → 筛选值得记的 → 提炼为简短 fact → 定位领域 → 检查冲突 → 写入/合并/更新 → 热度评估 → compact

### 性格线
原始记录 → 提取交互模式 → 积累信号 → 达到阈值 → 修改主 Agent 的 system prompt / 行为指令

性格不是"知识"，而是对主 Agent 行为的直接修改。

## 七、硬规则

### 时间绝对化
所有写入记忆的时间信息必须转为绝对时间：

```
❌ "用户下周要去东京出差"
✅ "用户 2026-03-14 ~ 2026-03-21 去东京出差（2026-03-07 提及）"
```

记录时间（何时提及）和事件时间（何时发生）都要保留。

### 写入权
只有 Memory Agent 可以写入记忆存储。主 Agent 不直接修改记忆文件。

### 随身索引
只放索引，不放内容。超限触发 compact，不触发丢弃。

### 记忆可信度（自述优先原则）

每条记忆必须标注来源和可信度。核心原则：**自述优先**——一个人关于自己说的话，可信度最高。

| 来源 | 关于谁 | 可信度 | 处理方式 |
|---|---|---|---|
| 用户自己说 | 关于自己 | 完全可信 | 直接写入 |
| 用户自己说 | 关于别人 | 较可信（用户视角） | 写入，标注"用户认为" |
| 别人说 | 关于用户 | 存疑 | 写入但标注来源，不覆盖用户自述 |
| 别人说 | 关于别人自己 | 较可信 | 写入，类似用户自述的逻辑 |
| 别人说 | 关于第三方 | 低可信 | 记录为"X 说 Y 怎样"，不当事实 |

冲突处理示例：
```
用户说："我没换工作"
群里有人说："Yuelun 不是跳槽了吗？"

✅ 以用户自述为准
✅ 记录为："某某认为用户换了工作（与用户自述矛盾，2026-03-07）"
❌ 不能用第三方说法覆盖用户自述
```

## 八、同步 vs 异步写入

同一条管道，粒度不同：

| | 同步（用户说"记住这个"） | 异步（定期整理） |
|---|---|---|
| 触发 | 主 Agent 转发给 Memory Agent | hook / 定时 |
| 筛选 | 跳过（用户已决定重要性） | 需要 LLM 判断什么值得记 |
| 定位 | 定位 + 写入 | 定位 + 写入 + 合并 |
| 热度 | 不需要评估 | 需要评估升降级 |
| 额外工作 | 无 | compact、性格演化、遗忘 |

## 九、用例推演

### 用例 1：跨会话事实记忆

**场景**：用户今天说"我下周去东京出差"。三周后说"好累啊"。

**流程**：
1. 对话中，这句话进入 → **原始记录**
2. Memory Agent 被动触发（对话结束 hook）→ 提炼 fact "用户 2026-03-14 ~ 2026-03-21 去东京出差"
3. 写入 → **检索知识/用户/近期动态**
4. 会话级收尾时生成 → **语义索引文件**（模糊印象层）
5. 判断为近期重要事项 → 升到 **随身索引**："用户 2026-03-14 ~ 2026-03-21 去东京出差"
6. 三周后用户说"好累"→ 主 Agent 随身索引里就有这条 → 直接脱口而出："是不是东京出差累到了？"
7. 再过两个月没提 → 热度衰减 → 从随身索引降回检索知识

### 用例 2：模糊回想

**场景**：用户问"我们是不是聊过 Qdrant？"

**流程**：
1. 主 Agent 扫随身索引 → 没有 Qdrant 相关
2. 主 Agent 问 Memory Agent："我们聊过 Qdrant 吗？"
3. Memory Agent grep 语义索引文件 → 命中 2026-02-20 的会话摘要（关键词里有 Qdrant）
4. Memory Agent **先快速回复**："好像有，是 2026-02-20 聊向量数据库选型时提过"
5. 主 Agent → "等等……让我想想……好像是有！"
6. Memory Agent 同时去读该会话的原始记录 → **补充细节**："当时你在比较 Qdrant 和 Milvus，最后没选"
7. 如果该话题跨了多次会话 → Memory Agent 拼接相关片段，给出完整上下文

### 用例 3：群聊

**场景**：用户把 Aria 拉进一个技术群，五个人打招呼。

**流程**：
1. 五个人的消息 → **原始记录**
2. Memory Agent 被动触发 → 在 **检索知识** 下创建 `群/技术群A/`，记录成员信息
3. 群主和高频发言者 → 热度高 → 升到 **随身索引**："技术群A：群主小明，常聊后端"
4. 偶尔冒泡的人 → 留在检索知识，被提到时 Memory Agent 现查

### 用例 4：性格适应

**场景**：用户连续二十次对话都用简短风格。

**流程**：
1. 每次对话 → **原始记录**
2. Memory Agent 异步性格线 → 分析交互模式 → "用户偏好简短对话"信号反复出现
3. 积累 20+ 次 → 达到阈值 → Memory Agent **修改主 Agent 的行为指令**：提高"回复简洁度"权重
4. 主 Agent 渐进地变得更简洁

### 用例 5：用户主动说"记住这个"

**场景**：用户说"记住，我的服务器 root 密码是不能告诉别人的"

**流程**：
1. 主 Agent 识别到"记住"意图 → 主动触发 Memory Agent
2. Memory Agent 跳过筛选（用户已声明重要性）
3. 转为绝对时间 → 写入 **检索知识/用户/安全规则**
4. **随身索引** 加一行："安全：服务器 root 密码不可外泄 [已记录]"
5. 不在随身索引里放密码本身

### 用例 6：记忆冲突

**场景**：半年前说"在学 Rust"，现在说"在写 Go，Rust 放弃了"

**流程**：
1. 新对话 → **原始记录**
2. Memory Agent 异步触发 → 提炼 fact "用户在写 Go"
3. 检测到与已有 fact "用户在学 Rust" 冲突
4. 更新 **检索知识**：标记旧条目为历史（"2025-12 ~ 2026-02 学过 Rust，后放弃"）
5. **随身索引** 更新："用户目前在写 Go"

### 用例 7：领域知识学习

**场景**：用户说"帮我把这篇 DeFi 借贷的文章学一下"

**流程**：
1. 这不走"经历 → 提炼"路线，而是 Memory Agent **直接写入检索知识** `知识/defi/借贷协议.md`
2. 会话收尾时生成 **语义索引文件**，包含该学习内容的关键词和概念
3. **随身索引** 加一行："领域知识：DeFi 借贷 [有详细笔记]"
4. 后续频繁聊 DeFi → 核心概念可能升到随身索引的摘要里
5. 长期不提 → 热度衰减，随身索引只保留一行索引，细节留在检索层

### 用例 8：随身索引 compact

**场景**：随身索引接近容量上限。

**流程**：
1. Memory Agent 检测到随身索引即将超限
2. 扫描每条的元数据（最后访问时间、访问次数）
3. 合并相关条目："用户在写 Go" + "用户之前学过 Rust" + "用户偏好静态类型" → "用户偏好静态类型语言，目前写 Go（曾学 Rust）"
4. 降级低热度条目 → 从随身索引移除，检索知识里还有
5. 精简表述 → 去掉冗余修饰词
6. 确保压缩后不丢失"我知道什么"的索引能力

## 十、已确定事项（本轮新增）

| 话题 | 结论 |
|---|---|
| 模糊印象实现 | LLM 生成描述性语义索引文件（按会话），字符串匹配，命中再展开原始记录 |
| 搜索流程 | 随身索引 → 语义索引文件 → 原始记录，Memory Agent 分步回复（先"好像有"再补细节） |
| 睡眠定义 | 用户手动告知会话结束 |
| 睡眠分级 | 会话级收尾（轻量：生成索引、提炼知识）+ 全局睡眠（重量：compact/自审/性格线） |
| 并发模型 | Memory Agent 单例，多会话共享记忆，知识自动跨会话流动 |
| 随身索引容量 | ~200 条，分区管理（用户/话题/提醒/近期上下文/备用） |
| 近因效应 | 随身索引有"近期上下文"专区，按会话分小区域，睡眠时刷新 |
| 索引自我修复 | Memory Agent 读会话记录时，发现记忆被召回但索引缺关键词 → 补充；发现误命中 → 修正 |
| 记忆可信度 | 自述优先原则：自己说自己的最可信，第三方转述标注来源、不覆盖自述 |
